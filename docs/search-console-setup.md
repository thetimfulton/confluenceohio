# Google Search Console & Webmaster Tools Setup

Setup guide for search engine verification, sitemap submission, and performance monitoring baseline for confluenceohio.org.

Spec: docs/12-seo-structured-data.md, docs/03-content-strategy-seo.md §6.3.

---

## 1. Domain Verification via Cloudflare DNS

Google Search Console requires domain ownership verification. We use DNS TXT records via Cloudflare since our DNS is managed there.

### Steps

1. Go to [Google Search Console](https://search.google.com/search-console/) and sign in with the project Google account.
2. Click **Add property** and choose **Domain** (not URL prefix). Enter `confluenceohio.org`.
3. Google provides a TXT record value like `google-site-verification=XXXXXXXXXXXXX`.
4. In the [Cloudflare dashboard](https://dash.cloudflare.com/), navigate to the `confluenceohio.org` zone > **DNS** > **Records**.
5. Add a TXT record:
   - **Type:** TXT
   - **Name:** `@` (root domain)
   - **Content:** The verification string from step 3
   - **TTL:** Auto
6. Back in Search Console, click **Verify**. DNS propagation may take up to 72 hours, but typically resolves within minutes via Cloudflare.
7. Copy the verification token (just the `XXXXXXXXXXXXX` part) and set it as the `GOOGLE_SITE_VERIFICATION` environment variable in Vercel:
   ```
   GOOGLE_SITE_VERIFICATION=XXXXXXXXXXXXX
   ```
   This populates the `<meta name="google-site-verification">` tag rendered in `apps/web/app/layout.tsx`.

### Bing Webmaster Tools (lower priority)

1. Go to [Bing Webmaster Tools](https://www.bing.com/webmasters/) and sign in.
2. Add `confluenceohio.org` and choose DNS verification.
3. Add the provided CNAME or TXT record in Cloudflare.
4. Set the `BING_SITE_VERIFICATION` env var in Vercel.

## 2. Sitemap Submission

Our sitemap is auto-generated by Next.js at `/sitemap.xml` (see `apps/web/app/sitemap.ts`). It includes:

- All 15 static routes with appropriate `changeFrequency` and `priority`
- Published MDX blog posts (`/blog/[slug]`)
- Approved voice submissions (`/voices/[slug]`)

### Submit to Google

1. In Search Console, select the `confluenceohio.org` property.
2. Go to **Sitemaps** in the left sidebar.
3. Enter `sitemap.xml` in the "Add a new sitemap" field and click **Submit**.
4. Verify status shows "Success" and the correct number of discovered URLs.

### Submit to Bing

1. In Bing Webmaster Tools, go to **Sitemaps**.
2. Submit `https://confluenceohio.org/sitemap.xml`.

### Validation

The sitemap is already referenced in `robots.txt` (see `apps/web/app/robots.ts`):

```
Sitemap: https://confluenceohio.org/sitemap.xml
```

Run the verification script to confirm the sitemap is valid:

```bash
npx tsx scripts/verify-seo.ts
```

## 4. Performance Monitoring Baseline

Establish a pre-launch baseline for the metrics we'll track post-launch.

### Core Web Vitals targets (from Artifact 12 §4)

| Metric | Target | Tool |
|--------|--------|------|
| LCP (Largest Contentful Paint) | < 2.5s | Lighthouse, Search Console, Vercel Analytics |
| INP (Interaction to Next Paint) | < 200ms | Search Console (after traffic), Vercel Analytics |
| CLS (Cumulative Layout Shift) | < 0.1 | Lighthouse, Search Console |
| TTFB (Time to First Byte) | < 800ms | Lighthouse |

### Lighthouse baseline

Run Lighthouse on the key pages and record scores. Targets:

| Category | Target |
|----------|--------|
| Performance | >= 90 |
| Accessibility | >= 95 |
| Best Practices | >= 90 |
| SEO | >= 95 |

```bash
# Install Lighthouse CLI if not available
npm install -g lighthouse

# Run against production (or local)
lighthouse https://confluenceohio.org --output=json --output-path=./lighthouse-homepage.json
lighthouse https://confluenceohio.org/sign --output=json --output-path=./lighthouse-sign.json
lighthouse https://confluenceohio.org/the-case --output=json --output-path=./lighthouse-the-case.json
```

### Bundle size targets

| Metric | Target |
|--------|--------|
| Initial page weight | < 500KB |
| JavaScript (compressed) | < 200KB |

Verify with bundle analyzer:

```bash
ANALYZE=true npm run build --filter=web
```

### Ongoing monitoring

After launch, monitor via:

1. **Search Console > Core Web Vitals report** — Aggregated field data from real users (requires traffic).
2. **Vercel Analytics** — Real User Monitoring (RUM) for LCP, INP, CLS, TTFB.
3. **PostHog** — Custom performance events if configured per Artifact 13.

## 5. Structured Data Validation

Before launch, validate all JSON-LD using Google's tools:

1. **Rich Results Test** — https://search.google.com/test/rich-results
   - Test each page type: homepage, blog post, voice article, FAQ

2. **Schema Markup Validator** — https://validator.schema.org/
   - Paste page URLs to validate @context, @type, required properties

3. **Automated validation** — Run the SEO verification script:

## 7. Troubleshooting

### Sitemap not found

- Verify the site is deployed and `/sitemap.xml` returns valid XML
- Check `robots.txt` references the correct URL
- Re-submit in Search Console

### Structured data errors

- Check `apps/web/lib/schema.ts` for the schema generators
- Verify the `<JsonLd>` component is rendered on the page
- Use browser DevTools to inspect `<script type="application/ld+json">` tags

### Pages not being indexed

- Check `robots.txt` is not blocking the path
- Verify the page does not have `<meta name="robots" content="noindex">`
- Use URL Inspection tool to see Google's rendered version
- Check for JavaScript rendering issues (SSR should handle this)

### OG images not appearing in shares

- Test with the [Facebook Sharing Debugger](https://developers.facebook.com/tools/debug/)
- Test with the [Twitter Card Validator](https://cards-dev.twitter.com/validator)
- Verify OG image URL is absolute and accessible
- Run: `npx tsx scripts/test-og-images.ts`
